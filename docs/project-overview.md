# Sloth Cloud 一期项目说明

## 1. 项目定位

`Sloth Cloud / 树懒云` 的目标不是“换一个 Paymenter 主题”，而是基于现有计费与服务基础设施，建立一个独立品牌、独立前端体验、独立 BFF 接口层的 VPS 服务商客户端。

一期聚焦客户可见层：

- 首页
- 商店/分类页
- 产品详情/购买配置页
- 客户服务详情页

## 2. 一期目标

- 建立独立前端品牌体验，不暴露 Paymenter token
- 用 `apps/api` 统一适配 Paymenter 管理 API
- 用 `apps/web` 消费规范化后的 BFF 数据
- 支持多语言，默认中文
- 支持深色/浅色主题，深色优先
- 先做可运行原型，后续逐步接真实业务逻辑

## 3. 品牌与体验方向

品牌调性建议：

- 专业、可信、克制，而不是炫技堆砌
- 科技感来自秩序、密度、信息设计，而不是霓虹模板
- 保持 VPS 服务商应有的“性能、稳定、全球节点、交付速度”表达

视觉方向建议：

- 深色主题为主，浅色主题作为企业感补充
- 主色建议走 `深空灰 + 冰川青 + 酸性绿` 路线
- 强调卡片层次、数据块、状态条、节点标签和即时反馈

## 4. 一期业务边界

本阶段明确不做：

- Proxmox 真实控制
- Convoy panel 真实控制
- 浏览器直连 Paymenter
- 复杂支付流程闭环
- 完整用户认证体系替换

本阶段明确要做：

- 目录结构与技术骨架
- 文档与接口约定
- BFF 的 mock/live 双模式
- 支持真实 Paymenter 商品/服务读取的适配入口
- 可演示的前台原型页面

## 5. 为什么要加 BFF

`apps/api` 作为 BFF 有几个必要性：

- 浏览器不直接持有 Paymenter token
- 统一把 Paymenter 的 JSON:API 数据转换为更适合前端消费的结构
- 对外暴露稳定的 Sloth Cloud 接口，避免前端被上游数据结构牵着走
- 后续接入 panel、Proxmox、工单、风控、营销活动时可以继续扩展
